## 一、项目背景与需求分析

### 1.1 项目背景

在当今高等教育环境中，学生面临着越来越多的学业挑战：

- **课程管理复杂**：需要管理多门课程的成绩、学分、GPA 等信息，数据分散在不同平台
- **目标规划困难**：缺乏系统化工具来制定和追踪学业目标
- **经历记录分散**：实习、项目、竞赛等经历没有统一的归档方式
- **简历制作繁琐**：每次写简历都需要从头整理信息，重复劳动
- **数据分析不足**：缺乏对自身学业数据的深度分析，无法直观看到进步趋势

### 1.2 需求分析

基于以上痛点，我确定了以下核心需求：

1. **全面的数据管理**：支持课程、角色、成果、经历、活动、目标、岗位等 7 类数据的增删改查
2. **智能数据分析**：提供 GPA 趋势分析、学期对比、同学对标等可视化功能
3. **简历自动生成**：基于已有数据智能生成简历，支持多格式导出
4. **AI 辅助分析**：通过规则引擎和远程 AI 提供学业建议
5. **数据导入导出**：支持 CSV 批量导入
6. **现代化界面**：Material Design 风格，流畅动画

> 【讲解提示】这一部分讲 1-2 分钟即可，重点是让评委理解"为什么要做这个系统"。可以举一个具体场景，比如："我曾经为了写简历，要翻遍手机相册、各个平台的记录去拼凑经历，这就是我做这个系统的初衷。"

### 2.2 系统架构

系统采用**六层分层架构**，从上到下依次为：

```
┌─────────────────────────────────────────────────────────────────┐
│                   表示层 (Presentation Layer)                     │
│   MainWindow → QStackedWidget → 12个Page + 9个Dialog            │
│   布局管理: QHBoxLayout / QVBoxLayout / QGridLayout / QSplitter  │
├─────────────────────────────────────────────────────────────────┤
│                     核心层 (Core Layer)                           │
│   AppShellController  ─ 壳层控制 (QStackedWidget 页切换)         │
│   DataRefreshCoordinator ─ 刷新协调 (信号槽跨页面同步)           │
│   BackendRuntimeController ─ 后端管理 (QProcess 启停)            │
│   AiContextMediator ─ AI上下文 (QEvent::eventFilter 事件过滤)    │
├─────────────────────────────────────────────────────────────────┤
│                    业务层 (Business Layer)                        │
│   Service 层: 13个服务类 (CourseService, AiService 等)           │
│   Model 层: 11个数据模型 (Course, Goal, Role 等)                 │
├─────────────────────────────────────────────────────────────────┤
│                     数据层 (Data Layer)                           │
│   DAO 层: 继承 DaoBase → QSqlDatabase + QSqlQuery               │
│   SQLite 数据库: schema.sql 定义表结构                           │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 核心协作关系：MainWindow 如何"指挥"各模块

这是整个系统最核心的设计。`MainWindow` 本身只有 **766 行**（从最初的 5000+ 行重构而来），它不直接处理业务逻辑，而是通过四个**核心协调器**来分发职责：

```
MainWindow (QMainWindow)
    │
    ├── m_shellController (AppShellController*)  ── 管理 QStackedWidget 页面切换
    │     └── 接收 NavigationListWidget 的 currentRowChanged 信号
    │         → 更新 topbar 标题、内容宽度、间距
    │
    ├── m_refreshCoordinator (DataRefreshCoordinator*) ── 管理跨页面数据刷新
    │     └── bindPages() 绑定全部 12 个页面
    │         connectSignals() 将每个页面的 dataChanged 信号连接到 refreshByDomain()
    │         → 实现"课程页改了数据 → 总览页自动刷新GPA"
    │
    ├── m_backendController (BackendRuntimeController*) ── 管理 HTTP 后端进程
    │     └── 使用 QProcess 启动/停止后端服务
    │         bindWidgets() 绑定 QProgressBar + QLabel + QSystemTrayIcon
    │         → 通过信号 backendReady / serverError 通知 MainWindow
    │
    └── m_aiMediator (AiContextMediator*) ── 管理 AI 上下文传递
          └── bindRootWidget() 在根 Widget 上安装 eventFilter
              重写 eventFilter() 拦截 QEvent::MouseButtonRelease
              → 检测用户选中文本 → 弹出浮窗菜单 → 发送到 AiPanelWidget
```

**页面注册机制**：MainWindow 使用 `std::function` 注册表替代了传统的 switch-case 硬编码：

```cpp
// 将每个页面添加到 QStackedWidget，并注册其刷新回调
m_overviewPage = new OverviewPage(this);
m_pageRefreshers[m_stack->addWidget(m_overviewPage)] = [this]() { 
    m_overviewPage->refresh(); 
};
// ... 12 个页面依次注册
```

这样在切换页面时，只需一行代码即可刷新：

```cpp
void MainWindow::refreshCurrentPage() {
    int index = m_stack->currentIndex();
    if (m_pageRefreshers.contains(index)) {
        m_pageRefreshers[index]();  // 调用对应页面的 refresh()
    }
}
```

> 【讲解提示】这是演讲的核心亮点之一。建议打开架构图的 PPT 页面，用手指着图讲解："MainWindow 就像一个'指挥官'，它不亲自干活，而是把任务分给四个协调器。比如用户在课程页修改了成绩，`DataRefreshCoordinator` 会自动通知总览页刷新 GPA 显示——这就是 Qt 信号槽机制的典型应用。" **这里可以现场演示：先切到课程页加一门课，再切回总览页看 GPA 变化。**

**侧边栏折叠动画**的关键实现：

```cpp
void SidebarWidget::setupToggleAnimation() {
    connect(m_toggleButton, &QPushButton::clicked, this, [this]() {
        auto* group = new QParallelAnimationGroup(this);
        auto* anim1 = new QPropertyAnimation(this, "minimumWidth");
        auto* anim2 = new QPropertyAnimation(this, "maximumWidth");
        anim1->setDuration(220);
        anim1->setEasingCurve(QEasingCurve::InOutQuad);
        // ... 同时对 minimumWidth 和 maximumWidth 做动画
        // 折叠时隐藏 brandMark 和 brandText
        // 展开时重新显示
        group->start();
    });
}
```

使用 `QParallelAnimationGroup` 让 `minimumWidth` 和 `maximumWidth` 两个属性**同时**做动画，确保宽度平滑过渡。`QEasingCurve::InOutQuad` 提供了"先慢后快再慢"的自然缓动感。

> 【讲解提示】侧边栏动画是一个很好的视觉演示点。可以现场点击折叠按钮，让评委看到流畅的动画效果。讲解时可以说："这里用了 Qt 的属性动画系统，`QPropertyAnimation` 可以对任何 QObject 的属性做动画，配合 `QParallelAnimationGroup` 实现多属性同步。"

**CrudPageController 的策略模式设计**：

这是项目中一个很巧妙的设计。`CrudPageController` 使用 `std::function` 回调实现了一个**通用的 CRUD 框架**，任何页面都可以复用：

```cpp
class CrudPageController : public QObject {
    // 策略函数类型定义
    using CreateDialogFn = std::function<QJsonObject(QWidget*)>;
    using EditDialogFn   = std::function<QJsonObject(QWidget*, const QJsonObject&)>;
    using ServiceCreateFn = std::function<void(const QJsonObject&)>;
    using ServiceUpdateFn = std::function<void(int, const QJsonObject&)>;
    using ServiceRemoveFn = std::function<void(int)>;
    // ...
    
    void executeAdd(QWidget* parent);    // 弹出创建对话框 → 调用 Service 创建
    void executeEdit(QWidget* parent, int id);  // 弹出编辑对话框 → 调用 Service 更新
    void executeRemove(QWidget* parent, int id); // 确认删除 → 调用 Service 删除
};
```

**以目标页面为例的完整数据流**：

```
用户点击"添加目标"
    → GoalsPage::onAddClicked()
    → GoalEditorDialog (QDialog) 弹出
        ├── 内含 QLineEdit (标题)
        ├── QComboBox (优先级: 高/中/低)
        ├── QDoubleSpinBox (目标值/当前值)
        ├── QPlainTextEdit (描述)
        └── validateAndAccept() 校验后返回 QDialog::Accepted
    → GoalService::create() 业务逻辑层处理
    → GoalDao::create() 数据访问层写入 SQLite
    → emit dataChanged(DataDomain::Goals) 发出信号
    → DataRefreshCoordinator::refreshByDomain() 
    → OverviewPage::refresh() 自动更新总览数据
```

**GoalEditorDialog 的控件布局**：

#### 3.5.2 AiContextMediator：事件过滤器实现"划词提问"

这是项目中最巧妙的设计之一。`AiContextMediator` 通过 Qt 的 **eventFilter 机制**，在全局范围内拦截鼠标事件，实现"在任意位置选中文本 → 弹出浮窗 → 发送到 AI"的功能。

```cpp
bool AiContextMediator::eventFilter(QObject* watched, QEvent* event) {
    // 1. 子控件添加时，自动让其文本可选
    if (event->type() == QEvent::ChildAdded) {
        makeTextSelectable(qobject_cast<QWidget*>(childEvent->child()));
    }
    
    // 2. 鼠标按下：记录位置
    if (event->type() == QEvent::MouseButtonPress) {
        m_leftMousePressed = true;
        m_mousePressGlobalPos = mouseEvent->globalPosition().toPoint();
    }
    
    // 3. 鼠标移动：判断是否拖拽足够距离（区分点击和选中）
    if (event->type() == QEvent::MouseMove && m_leftMousePressed) {
        int distance = (pos - m_mousePressGlobalPos).manhattanLength();
        if (distance >= QApplication::startDragDistance())
            m_draggedEnoughForSelection = true;
    }
    
    // 4. 鼠标释放：检测选中文本，弹出浮窗菜单
    if (event->type() == QEvent::MouseButtonRelease) {
        QTimer::singleShot(50, this, [this]() {  // 延迟50ms等Qt完成选择
            QString selectedText = selectedTextFromWidget(QApplication::focusWidget());
            if (!selectedText.isEmpty() && selectedText.length() > 1)
                showFloatingMenu(QCursor::pos(), selectedText);
        });
    }
}
```

**为什么用 eventFilter 而不是继承？** 因为我们需要在**所有**子控件（QLabel、QListWidget、QTableWidget 等）上都能检测选中文本，如果用继承就要改每个控件的基类，而 eventFilter 只需在根 Widget 上安装一次就能拦截所有子控件的事件。

#### 3.5.3 AiService：双模式 AI 引擎

`AiService` 是一个静态工具类，支持两种工作模式：

```
AiService::chat(data)
    │
    ├── 检测 isAiServerAvailable() 
    │     └── 使用 QNetworkAccessManager::get() 请求 /health 端点
    │         QEventLoop 同步等待，QTimer::singleShot 设置 2000ms 超时
    │         缓存结果 15 秒（可用）或 3 秒（不可用）避免频繁探测
    │
    ├── 远程模式 (Remote)
    │     └── callAiServer() → QNetworkAccessManager::post() 
│         携带 JSON body，QJsonDocument::fromJson() 解析响应
    │
    └── 本地规则模式 (Local)
          └── chatLocal() → generateLocalReply()
              analyzeLocal() → 
                buildCourseAdvice()   // 基于课程数据生成建议
```

```
    → 信号 chatMessageSent(const QString& message)
```

## 四、技术难点与解决方案

### 4.1 MainWindow 从 5000 行重构到 766 行

**难点**：原始 MainWindow 承载了所有逻辑——页面创建、数据刷新、后端管理、AI 集成、UI 布局，代码超过 5000 行，可维护性极差。

**解决方案**：**职责分离 + 四大协调器**

| 职责 | 原始位置 | 重构后 | 核心 Qt 类 |
|------|---------|--------|-----------|
| 页面切换控制 | MainWindow::onNavigationChanged() | `AppShellController` | `QStackedWidget`, `QLabel` |
| 数据刷新协调 | 各页面手动调用 refresh() | `DataRefreshCoordinator` | `QObject::connect()` 信号槽 |
| 后端进程管理 | MainWindow 内嵌 QProcess | `BackendRuntimeController` | `QProcess`, `QProgressBar` |
| AI 上下文传递 | MainWindow::eventFilter() | `AiContextMediator` | `QEvent`, `QMouseEvent` |

另外，页面注册使用 `QMap<int, std::function<void()>>` 替代 switch-case，新增页面只需一行注册代码。

> 【讲解提示】重构部分是很好的技术深度展示。可以说："最初我把所有代码都写在 MainWindow 里，很快就发现维护不动了。于是我按照'单一职责原则'，把不同功能拆分到四个独立的协调器中。重构后 MainWindow 只剩 766 行，只负责'组装'，不负责'干活'。" **建议用代码对比的方式展示重构前后的差异。**

### 4.2 跨页面数据一致性

**难点**：7 个数据管理页面各自独立操作数据库，修改一个页面的数据后，其他相关页面（如总览页、分析页）需要同步更新。

**解决方案**：**DataDomain 枚举 + 信号槽自动刷新**

```cpp
// DataDomain.h - 定义数据域枚举
enum class DataDomain {
    Courses, Roles, Achievements, Experiences, 
    Activities, Goals, Jobs, Analysis, All
};

// 每个页面发出 dataChanged 信号
signals:
    void dataChanged(DataDomain domain);

// DataRefreshCoordinator 统一监听
void DataRefreshCoordinator::connectSignals() {
    connect(m_courses, &CoursesPage::dataChanged, this, &DataRefreshCoordinator::refreshByDomain);
    connect(m_goals, &GoalsPage::dataChanged, this, &DataRefreshCoordinator::refreshByDomain);
    // ... 所有页面都连接
}

void DataRefreshCoordinator::refreshByDomain(DataDomain domain) {
    // 总是刷新总览页和时间轴页
    m_overview->refresh();
    m_timeline->refresh();
    
    // 根据域刷新相关页面
    switch (domain) {
        case DataDomain::Goals:
            m_goals->refresh();
            m_analysis->refresh();
            m_resume->refresh();
            break;
        // ...
    }
}
```

> 【讲解提示】数据一致性可以用一个具体场景讲解："假设用户在目标页添加了一个新目标，这时总览页的目标进度、分析页的建议、简历页的素材推荐都需要更新。`DataRefreshCoordinator` 通过 Qt 的信号槽机制，自动完成这个联动——用户完全无感知。"

### 4.3 AI 双模式引擎与优雅降级

**难点**：远程 AI 服务可能不可用（未启动、网络问题），如何保证系统始终能提供 AI 功能？

**解决方案**：**三层降级策略**

```
AiService::chat(data)
    │
    ├── 第1层：检查远程 AI 可用性 (缓存15秒)
    │     isAiServerAvailable() → QNetworkAccessManager::get("/health")
    │     QEventLoop + QTimer::singleShot(2000ms) 超时保护
    │
    ├── 第2层：远程 AI 可用 → callAiServer()
    │     QNetworkAccessManager::post() 发送 JSON 请求
    │     QJsonDocument 解析响应
    │
    └── 第3层：远程不可用 → 本地规则引擎
          chatLocal() → generateLocalReply()
          analyzeLocal() → buildCourseAdvice() / buildGoalAdvice() / ...
```

**AiStatusBar** 实时显示当前模式（本地规则 / 远程 AI / 连接中），用户始终知道 AI 的工作状态。

> 【讲解提示】这部分可以现场演示：(1) 启动远程 AI 服务，展示"远程 AI 模式"；(2) 关闭远程服务，展示自动切换到"本地规则模式"；(3) 观察 AiStatusBar 的状态变化。可以说："这种'优雅降级'的设计思想来自实际工程经验——系统不应该因为一个外部服务不可用就完全瘫痪。"

### 4.4 事件过滤器实现全局文本选中

**难点**：需要在所有类型的控件（QLabel、QListWidget、QTableWidget、QTextEdit 等）上都能检测用户选中的文本，但这些控件的基类各不相同。

**解决方案**：**eventFilter 模式 + 递归 makeTextSelectable**

```cpp
void AiContextMediator::bindRootWidget(QWidget* root) {
    m_rootWidget = root;
    m_rootWidget->installEventFilter(this);  // 在根 Widget 安装事件过滤器
    makeTextSelectable(m_rootWidget);         // 递归设置子控件文本可选
}

bool AiContextMediator::eventFilter(QObject* watched, QEvent* event) {
    // ChildAdded 事件：新添加的子控件也自动设为可选
    if (event->type() == QEvent::ChildAdded) {
        makeTextSelectable(qobject_cast<QWidget*>(childEvent->child()));
    }
    // MouseMove + MouseButtonRelease：检测拖拽选中 → 提取文本 → 弹出浮窗
    // ...
}
```

**为什么用 manhattanLength() 判断距离？** 因为曼哈顿距离计算只需要加法和绝对值，比欧氏距离（需要开方）快得多，足以判断"用户是否真的在拖拽选中文字"。

> 【讲解提示】eventFilter 是 Qt 中一个高级但非常实用的特性。可以这样解释："普通的信号槽只能连接到特定对象的特定信号，但 eventFilter 可以拦截任何 QObject 的任何事件。我在根 Widget 上安装了过滤器，就像在整个窗口上铺了一层'透明薄膜'，所有鼠标事件都会先经过我的处理。"

> QEventLoop loop;
> QNetworkReply* reply = manager.get(request);
> QTimer::singleShot(kAiHealthTimeoutMs, &loop, &QEventLoop::quit);
> QObject::connect(reply, &QNetworkReply::finished, &loop, &QEventLoop::quit);
> loop.exec();  // 阻塞主线程，但有超时保护
> ```
>
> 当时为了快速实现 AI 双模式的**同步降级逻辑**（先探测远程是否可用，不可用则立即切换到本地），采用了这种同步写法。超时 2 秒是可接受的折中——用户只会感觉到一次短暂的'思考'。
>
> 但如果要优化到生产级别，我会做以下改造：
> 1. 将 `AiService` 的网络调用改为**纯异步模式**——`QNetworkAccessManager::get()` 本身就是异步的，只需连接其 `finished` 信号到一个回调槽函数
> 2. 在等待期间显示 `QProgressBar` 或加载动画，而不是冻结 UI
> 3. 或者使用 `QtConcurrent::run()` 将整个 AI 调用放入 `QThreadPool`，通过信号槽将结果传回主线程更新 UI"

**追问升级**：如果老师继续问"那你为什么不一开始就用异步？"

> "因为异步模式会让代码复杂度显著增加——需要处理回调嵌套、生命周期管理（如果用户在 AI 响应回来之前关闭了页面怎么办）、以及错误重试的状态机。在课程项目的时间约束下，我选择了'先跑通再优化'的策略。这也是软件工程中'迭代开发'思想的体现。"

#### Q2：C++ 内存管理（必问）

**老师可能这样问**：
> "C++ 最怕内存泄漏，你这 1.5 万行代码里 `new` 了这么多对象（动画、对话框、Widget），你是怎么管理内存的？有没有用智能指针？"

**标准回答**：

> "本项目主要依赖 **Qt 的对象树（Object Tree）机制**来管理内存。
>
> 所有继承自 `QObject` 的类——包括 Widget、Controller、Animation——在创建时都传入了 `this` 作为 parent 参数：
>
> ```cpp
> m_sidebar = new SidebarWidget(this);        // parent = MainWindow
> m_shellController = new AppShellController(this);  // parent = MainWindow
> anim1->setParent(this);                      // 动画对象挂到 Widget 下
> ```
>
> 当 `MainWindow` 在析构函数中销毁时，Qt 会**递归销毁**其所有子对象，这是 `QObject` 的内置机制，不需要手动 `delete`。
>
> 对于**非 QObject 的临时对象**（如 `QJsonObject`、`QJsonDocument`），它们是栈上对象或值类型，作用域结束自动销毁。
>
> 对于**需要手动管理的特殊对象**，我严格遵守了 Qt 的规范：
> - `QNetworkReply` 使用后立即调用 `reply->deleteLater()`（在 `AiService` 中有 3 处）
> - `QParallelAnimationGroup` 动画完成后通过 `connect(group, &QParallelAnimationGroup::finished, group, &QObject::deleteLater)` 自动释放
> - `ToastNotification` 淡出动画结束后自动调用 `deleteLater()` 销毁自身
> - 浮窗菜单 `m_floatingWidget` 在隐藏时调用 `deleteLater()`
>
> 项目中**没有使用 `std::unique_ptr` 或 `std::shared_ptr`**，因为 Qt 的对象树机制已经足够，混用智能指针反而可能导致'双重释放'问题——这是 Qt 官方文档明确建议的。"

**加分点**：如果老师问到 `deleteLater()` 和 `delete` 的区别：

> "`deleteLater()` 不是立即删除，而是向事件循环发送一个 `DeferredDelete` 事件，等当前事件处理完毕后再删除。这避免了在信号槽调用链中删除正在使用的对象导致的崩溃。比如 `QNetworkReply` 的 `finished` 信号回调中，reply 对象可能还在被 Qt 内部使用，所以必须用 `deleteLater()` 而不是直接 `delete`。"

### 二、AI 使用与原创性验证（核心必答题）

> **特别说明**：这是答辩中**最敏感、最可能被追问**的话题。老师问这个问题的目的不是要"抓你"，而是想看你的**诚实态度**和**技术判断力**。以下回答策略的核心思路是：**坦诚承认 AI 辅助 + 用具体例子划清边界 + 强调架构决策是自己的**。

#### Q3-A：你用了 AI 吗？用了多少？

**老师可能这样问**：
> "15000 行代码、151 个文件，一个人一学期是怎么写完的？你用了 AI 代码生成工具吗？"

**标准回答**：

> "老师，我确实在开发过程中使用了 AI 工具。但我对 AI 的使用有明确的边界和方法论——我把它称为**'原型驱动 + AI 辅助生成'**的开发模式。
>
> 具体来说：
> 1. 我**亲自设计并手写**了一个标准的模板类（比如 `CourseDao`），确立命名规范、内存管理方式和核心接口
> 2. 然后利用 AI 工具，输入不同数据表的字段定义，让 AI 基于我的模板**批量生成**其他同质化的类（如 `GoalDao`、`RoleDao` 等）
> 3. 最后我对 AI 生成的代码进行 **Code Review（代码审查）** 和微调
>
> 这种做法让我从机械的重复劳动中解放出来，将核心精力聚焦于**系统架构设计、协调器模式的应用以及复杂业务逻辑的实现**上。"

> 【讲解提示】**千万不要说"我让 AI 帮我复制改写"**——这会让老师觉得你在偷懒。用"原型驱动 + AI 辅助生成"这个说法，体现了你有方法论、有流程管控。老师听到"Code Review"和"原型模板"这些词，会认为你具备工程化思维。

**话术升级对照表**：

| 危险说法（不要说） | 专业说法（建议说） |
|---|---|
| "Model 层差不多，让 AI 复制改写" | "我采用原型驱动 + AI 辅助生成的开发模式" |
| "为了省事让 AI 写的" | "为了提高开发效率并保证代码规范" |
| "AI 写完我没怎么改" | "AI 生成后我负责 Code Review 和微调" |
| "这些代码没什么技术含量" | "这些是结构相似的样板代码（Boilerplate Code）" |

**理论支撑——如果老师追问"为什么这么做"，可以用以下三点回应**：

> 1. **符合 DRY 原则（Don't Repeat Yourself）**：人类开发者不应该把时间浪费在机械地敲击重复的 CRUD 代码上。利用 AI 生成样板代码，是现代开发中消除重复劳动的最佳实践。
>
> 2. **关注点分离（Separation of Concerns）**：我的核心竞争力在于架构设计（六层架构、协调器、事件过滤器），而不是"能多快写完 10 个 DAO 类的 SQL 语句"。让 AI 处理底层重复代码，我专注于高层架构——这是**架构师的思维**。
>
> 3. **现代开发范式（AI-Assisted Development）**：在工业界，使用 GitHub Copilot 或大模型生成 DTO、DAO 和单元测试已经是标准操作。我在这个项目中提前实践了这种**人机协同的开发流程**。

#### Q3-B：AI 具体帮你写了什么？（举具体例子）

**老师可能这样问**：
> "能具体说说哪些代码是 AI 帮你写的吗？"

**标准回答——用"三类代码"来分类说明**：

> "我把代码分成三类，AI 的参与程度完全不同："

**第一类：纯样板代码（AI 辅助率 ~80%，但价值低）**

> "这类代码模式高度重复，AI 能快速生成，但我需要逐行审查和调整。"
>
> **例子 1：DAO 层的 SQL 操作代码**
>
> 项目有 10 个 DAO 类（`CourseDao`、`GoalDao`、`RoleDao` 等），每个都有 `getAll()`、`getById()`、`create()`、`update()`、`remove()` 五个方法，结构几乎一样——只是表名和字段不同。比如 `CourseDao::create()`：
>
> ```cpp
> bool create(const Course& course) {
>     QSqlQuery query(m_db);
>     query.prepare("INSERT INTO courses (name, code, credits, ...) "
>                   "VALUES (:name, :code, :credits, ...)");
>     query.bindValue(":name", course.name);
>     query.bindValue(":code", course.code);
>     query.bindValue(":credits", course.credits);
>     // ... 12 个字段的绑定
>     return query.exec();
> }
> ```
>
> 这种代码 AI 可以根据 `Course` 模型类快速生成，但**字段名、绑定顺序、SQL 语法**我都要逐一检查。
>
> **例子 2：UI 布局代码**
>
> 比如 `CoursesPage::setupUi()` 中大量的 `new QLabel`、`addWidget`、`setContentsMargins`、`setObjectName` 调用，这类代码没有逻辑，纯粹是"体力活"，AI 生成效率很高。

**第二类：业务逻辑代码（AI 辅助率 ~40%，需要深度修改）**

> "这类代码有明确的业务逻辑，AI 能提供初始框架，但我需要大量修改。"
>
> **例子 1：`CourseService::getStatistics()` 的 GPA 计算逻辑**
>
> AI 生成了基本的遍历和求和代码，但以下逻辑是我自己设计的：
> - GPA 的两种计算标准（`standard` 4.0 制 vs `4.33` 制）的选择逻辑
> - 学分进度的百分比计算和上限封顶（`std::min(..., 100.0)`）
> - 按学期分组统计的 `QMap<QString, QList<Course>>` 数据结构选择
>
> **例子 2：`AiService` 的双模式切换逻辑**
>
> AI 帮我写了 `QNetworkAccessManager` 的基本调用代码，但**双模式切换的核心设计**是我自己的：
> - 健康检查的缓存策略（可用时缓存 15 秒，不可用时 3 秒后重试）
> - 本地规则引擎的 6 种建议类型（课程/目标/经历/简历/成就/综合）
> - 优雅降级的三层策略（远程可用 → 远程调用 → 本地规则）

**第三类：架构设计代码（AI 辅助率 ~0%，完全自主设计）**

> "这类代码是系统的'骨架'，AI 无法替代——因为它需要对整个系统有全局理解。"
>
> **例子 1：`CrudPageController` 的策略模式设计**
>
> 这个类的核心设计决策——用 `std::function` 回调注入不同页面的具体逻辑——是我自己想出来的。AI 不知道我有 7 个 CRUD 页面需要复用同一套增删改查流程，也不知道我需要用 `QJsonObject` 作为中间数据格式来统一不同数据类型。
>
> **例子 2：`AiContextMediator` 的 eventFilter 设计**
>
> 我需要在所有类型的控件上实现"划词提问"功能。这个需求的解决方案——用 `eventFilter` 拦截全局鼠标事件、用 `qobject_cast` 逐类型检测选中文本、用 `manhattanLength` 判断拖拽距离——是我自己设计的。AI 可以帮我写 `eventFilter` 的框架代码，但它不知道为什么要用 `manhattanLength` 而不是欧氏距离，也不知道为什么要处理 `QEvent::ChildAdded` 来支持动态添加的控件。
>
> **例子 3：`DataRefreshCoordinator` 的信号槽拓扑**
>
> 12 个页面的数据刷新联动，如果两两连接信号槽会产生 132 条连接线。我设计了星型拓扑——所有页面只发出 `dataChanged(DataDomain)` 信号，协调器集中维护刷新规则。这个**架构决策**需要对整个系统的数据流有全局理解，AI 无法替代。
>
> **例子 4：从 5000 行重构到 766 行**
>
> 重构的核心工作是**识别职责边界**——哪些代码属于"壳层控制"（拆到 `AppShellController`）、哪些属于"数据刷新"（拆到 `DataRefreshCoordinator`）、哪些属于"后端管理"（拆到 `BackendRuntimeController`）。这需要理解每段代码的**依赖关系**和**变化原因**，是纯粹的架构思考，不是代码生成能解决的。

#### Q3-C：你怎么保证 AI 生成的代码是正确的？

**老师可能这样问**：

**老师可能这样问**：

**标准回答**：

#### Q3-F：AI 生成的代码有 Bug 你怎么发现？

**老师可能这样问**：
> "AI 生成的代码如果有一处隐蔽的 Bug——比如字段类型映射错了，或者内存泄漏——你怎么发现？"

**标准回答**：

> "老师，这也是我坚持进行 Code Review 的原因。AI 只是生成器，我是最终的把关人。

> **第一道防线：编译期检查**。C++ 是强类型语言，如果 AI 把 `int` 错写成 `QString`，或者 Qt 的信号槽签名不匹配，编译器（MSVC/GCC）会直接报错，无法通过编译。这是静态类型语言相比 Python 等动态语言的优势。

>
> 举一个实际的例子：AI 生成的 `AiService` 代码最初没有在 `QNetworkReply` 使用后调用 `deleteLater()`，我在 Code Review 时发现了这个问题并手动补上——因为我理解 Qt 的内存管理机制。"

#### Q3-G：为什么不用 ORM 或万能基类来避免重复代码？

**老师可能这样问**：

**标准回答**：

> "老师，我确实评估过这两种方案，但最终选择了'独立类 + AI 辅助生成'，原因有两点：
>
> **第一，C++ 缺乏原生反射机制**。不像 Java 或 C# 有强大的反射，C++ 要实现通用的 ORM 往往需要引入宏定义或复杂的模板元编程，这会大大增加编译时间和代码复杂度。Qt 的元对象系统（`Q_OBJECT`）虽然提供了部分反射能力，但它的设计初衷是支持信号槽，不是用来做数据库映射。
>
> **第二，性能与可控性**。这个系统涉及多表查询和复杂统计——比如 `CourseService::getStatistics()` 中的按学期分组、GPA 加权计算、学分进度百分比等。手写具体的 SQL 语句虽然代码量大，但性能最高、SQL 调优最灵活。
>
> 所以我选择了**扁平化的独立类 + AI 辅助生成**的方案，这是在开发效率、运行性能和代码可控性之间的最佳权衡。"

### 三、架构与设计模式深度追问

#### Q4：SQLite 并发与锁问题

**老师可能这样问**：
> "你用了 SQLite，如果 HTTP Server 收到并发请求，同时 UI 也在读写数据库，会不会报 `database is locked`？"

**标准回答**：

> "这是一个很好的问题。目前系统主要是**单用户桌面端场景**，并发量极低——UI 线程操作数据库时，HTTP Server 的请求也会排队处理。
>
> 我在 `schema.sql` 中设置了 `PRAGMA foreign_keys = ON` 来保证数据完整性。SQLite 默认使用**文件锁**模式，写操作会阻塞其他读写。
>
> 如果要支持更高的并发，我了解到可以做以下优化：
> 1. 开启 **WAL (Write-Ahead Logging) 模式**：`PRAGMA journal_mode=WAL;`，这样读操作不会被写操作阻塞
> 2. 将所有数据库写操作放入**事务**中（`BEGIN TRANSACTION ... COMMIT`），减少锁的持有时间
> 3. 为 HTTP Server 的每个请求创建**独立的数据库连接**（Qt 的 `QSqlDatabase::addDatabase()` 支持命名连接）
>
> 但在当前的单用户桌面场景下，SQLite 的默认行为已经完全满足需求。"

**注意**：如实回答——项目中确实没有设置 WAL 模式，但你知道这个知识点。

#### Q5：协调器模式与中介者模式的区别

**老师可能这样问**：
> "你提到了 `DataRefreshCoordinator`（协调器）和 `AiContextMediator`（中介者），这两个模式有什么区别？为什么不直接用信号槽互相连接？"

**标准回答**：

> "这两个模式解决的是**不同层面的解耦问题**。
>
> **直接用信号槽的问题**：如果 12 个页面两两连接信号槽，会产生 N×(N-1) 条连接线，形成'蜘蛛网'式的依赖关系。每新增一个页面，就要和所有已有页面建立连接。
>
> **`DataRefreshCoordinator`（协调器）**解决的是**数据刷新的一对多广播问题**。它是一个**星型拓扑的中心节点**——所有页面只负责发出 `dataChanged(DataDomain)` 信号，不关心谁需要刷新；协调器集中维护'哪个域变化了应该刷新哪些页面'的映射规则。这符合**迪米特法则（最少知识原则）**。
>
> **`AiContextMediator`（中介者）**解决的是**不同类控件与 AI 面板之间的通信问题**。它通过 `eventFilter` 拦截全局鼠标事件，检测文本选中，然后转发给 `AiPanelWidget`。如果没有这个中介者，每个需要支持'划词提问'的控件（QLabel、QListWidget、QTableWidget 等）都要自己实现选中检测和 AI 通信逻辑。
>
> 简单来说：
> - 协调器 = **消息路由器**（'数据变了' → '谁需要知道'）
> - 中介者 = **协议转换器**（'鼠标事件' → 'AI 上下文'）"

---

#### Q6：为什么要用 eventFilter 而不是继承？

**老师可能这样问**：
> "你想让所有控件都支持文本选中，为什么不写一个 `SelectableWidget` 基类让所有控件继承？"

**标准回答**：

> "因为项目中的控件**基类各不相同**——有的继承 `QLabel`，有的继承 `QListWidget`，有的继承 `QTableWidget`，它们的共同祖先只有 `QWidget`。如果用继承方案，要么写多个 `SelectableLabel`、`SelectableListWidget`、`SelectableTableWidget`，要么用**多重继承**（这在 Qt 中会破坏 `Q_OBJECT` 宏的元对象系统）。
>
> 而 `eventFilter` 只需要在**根 Widget 上安装一次**，就能拦截其所有子控件的事件，包括未来动态添加的控件。代码量从'每个控件类改一遍'变成了'一处代码解决所有问题'。
>
> 而且我还处理了 `QEvent::ChildAdded` 事件——当有新的子控件被动态添加时，自动调用 `makeTextSelectable()` 让它也支持文本选中。这在继承方案中很难做到。"

### 四、Qt 技术细节追问

#### Q7：信号槽的连接类型

**老师可能这样问**：
> "你代码里的 `connect` 都是默认的 `AutoConnection`，如果信号和槽在不同线程怎么办？"

**标准回答**：

> "目前项目中所有信号和槽都在**主线程（UI 线程）**中，所以使用默认的 `Qt::AutoConnection` 就等同于 `Qt::DirectConnection`——信号发射时直接调用槽函数。
>
> 如果未来将 AI 调用放到后台线程，信号和槽就会跨线程，此时 `AutoConnection` 会自动变为 `Qt::QueuedConnection`——信号发射时将事件投递到接收者所在线程的事件队列中，由目标线程的事件循环来调用槽函数。这保证了**线程安全**——UI 更新永远在主线程执行。
>
> 我了解 Qt 还支持 `Qt::BlockingQueuedConnection`（阻塞等待目标线程处理完毕）和 `Qt::UniqueConnection`（避免重复连接），但在当前项目中没有使用场景。"

#### Q8：Q_OBJECT 宏的作用

**老师可能这样问**：
> "你的每个头文件都有 `Q_OBJECT` 宏，它到底做了什么？不写会怎样？"

**标准回答**：

> "`Q_OBJECT` 宏展开后会做三件事：
> 1. 声明 `staticMetaObject` 静态成员——这是 Qt **元对象系统**的核心数据结构
> 2. 声明 `qt_metacall()` 虚函数——信号槽机制通过它来调用槽函数
> 3. 声明 `tr()` 和 `metaObject()` 等方法
>
> 如果不写 `Q_OBJECT`：
> - **信号槽仍然可以编译通过**（因为 `connect` 是模板函数），但**运行时不会正确调用槽函数**——因为缺少元对象信息，Qt 无法通过字符串匹配找到槽函数
> - **无法使用 `qobject_cast`** 进行安全的类型转换
> - **无法使用 Qt 属性系统**（`Q_PROPERTY`）
>
> 所以我的项目中，所有需要使用信号槽的类（Widget、Controller、Mediator 等）都加上了 `Q_OBJECT` 宏。"

#### Q9：QSettings 存了什么？数据存在哪里？

**老师可能这样问**：
> "你用了 `QSettings` 做持久化，它和 SQLite 有什么区别？存了什么数据？"

**标准回答**：

> "`QSettings` 是 Qt 提供的**轻量级键值对存储**，适合保存少量配置信息。在 Windows 上它底层使用**注册表**，在 macOS 上使用 **plist 文件**，在 Linux 上使用 **INI 文件**。
>
> 本项目中 `QSettings` 存储的数据包括：
> - **窗口几何信息**：`settings.value("geometry")` 和 `settings.setValue("geometry", saveGeometry())`——记住窗口大小和位置
> - **学生个人信息**：姓名、学号、头像路径等——通过 `StudentInfoCard` 读写
> - **简历配置**：头像路径 `settings.value("profile/avatar")`
>
> 而 **SQLite** 存储的是**结构化业务数据**——课程、目标、经历等大量记录。
>
> 简单来说：`QSettings` 存'配置'，SQLite 存'数据'。"

### 五、工程实践追问

#### Q10：CMake 构建系统是怎么配置的？

**老师可能这样问**：
> "你的项目用了 CMake，能说说 `CMakeLists.txt` 里是怎么配置 Qt 模块的吗？"

**标准回答**：

> "在 `CMakeLists.txt` 中，我使用了 Qt 6 的 `find_package` 来查找所需的模块：
>
> ```cmake
> find_package(Qt6 REQUIRED COMPONENTS Core Widgets Sql Network)
> ```
>
> 然后用 `qt_add_executable` 创建可执行目标，并通过 `target_link_libraries` 链接这些模块。Qt 6 的 CMake 集成会自动处理 **MOC（元对象编译器）**——当头文件中有 `Q_OBJECT` 宏时，CMake 会自动调用 `moc` 生成元对象代码，不需要手动运行。
>
> 项目还提供了 `build_desktop.bat` 一键构建脚本，封装了 `cmake -B build && cmake --build build` 的完整流程。"

#### Q11：错误处理是怎么做的？

**老师可能这样问**：
> "你的代码里有没有做错误处理？比如数据库写入失败、网络请求超时怎么办？"

**标准回答**：

> "我在多个层面做了错误处理：
>
> **数据库层**：
> - `DaoBase` 构造时检查 `m_db.isOpen()`，如果未打开则尝试重新打开
> - `main_desktop.cpp` 中每个 SQL 语句执行后都检查 `query.exec()` 返回值，失败时记录 `Logger::warning`
> - 数据库初始化失败时弹出 `QMessageBox::critical` 并退出程序
>
> **网络层**：
> - `AiService` 的 `QNetworkReply` 检查 `reply->error()` 是否为 `QNetworkReply::NoError`
> - 设置了 `request.setTransferTimeout()` 超时保护
> - 健康检查有 2000ms 超时，AI 生成有 180000ms 超时
> - 网络失败时自动降级到本地规则引擎
>
> **UI 层**：
> - 删除操作前弹出 `QMessageBox::question` 确认对话框
> - 对话框有 `validateAndAccept()` 校验逻辑，输入不合法时阻止提交
> - 操作完成后通过 `ToastNotification::display()` 给用户反馈
>
> **日志系统**：
> - 自定义 `Logger` 类，支持 `info`、`warning`、`error` 三个级别
> - 关键操作都有日志记录，便于排查问题"
