# 团队成员分工说明

## 1. 文档目的

本文档用于说明《个人发展规划系统》在期中设计、期末实现与最终提交过程中的团队分工情况，重点对应课程要求中的“明确每位同学的实际分工与贡献”。本项目最终提交形态为 **Qt 原生 C++ 桌面应用**，采用分层架构组织代码，保留本地 HTTP API 作为调试、联调和后续扩展接口。

## 2. 总体分工原则

团队采用“组长统筹架构与界面，组员分别负责接口、业务数据层、测试文档”的方式推进。分工不是完全割裂的模块开发，而是围绕同一套架构进行协作：组长负责整体技术路线、模块边界和 Qt 原生界面实现；组员 A 负责 API 层与路由；组员 B 负责数据模型、DAO 与 Service 业务逻辑；组员 C 负责测试、文档、运行验证和提交材料整理。

## 3. 成员分工总览

| 成员 | 主要职责 | 对应目录/文件 | 工作量占比 |
|---|---|---|---|
| 组长 | 架构设计、技术选型、Qt UI 页面与交互、核心协调器、进度管理 | `src/client/`, `src/client/core/`, `src/client/pages/`, `src/client/widgets/`, `src/client/dialogs/`, `MainWindow` 相关代码 | 30% |
| 组员 A | API 层开发、HTTP 路由设计、请求响应封装、JSON 工具与接口联调 | `src/api/`, `src/server/`, `src/util/JsonUtils*` | 25% |
| 组员 B | 数据模型、DAO 层、Service 层、数据库表结构与业务规则 | `src/model/`, `src/dao/`, `src/service/`, `resources/schema.sql` | 30% |
| 组员 C | 测试用例、运行验证、日志系统、文档整理、提交包检查 | `tests/`, `docs/`, `README.md`, 报告与技术解析文档 | 15% |

## 4. 组长分工说明

### 4.1 架构设计与技术路线

组长负责确定项目从早期 Web 方案迁移为 Qt 原生 C++ 桌面应用的技术路线，明确系统采用“表现层、协调层、业务层、数据访问层、模型层、工具层”的分层结构。组长同时负责划分客户端页面、可复用组件、核心协调器、业务服务和数据访问模块之间的边界，避免所有逻辑堆叠在单一窗口类中。

### 4.2 Qt 原生界面与交互实现

组长负责 Qt UI 的整体风格和交互设计，包括左侧导航栏、主内容区、右侧 AI 助手栏、简历编辑器、弹窗编辑器、Toast 提示、页面切换和布局响应。该部分主要对应：

| 模块 | 说明 |
|---|---|
| `src/client/pages/` | 总览、课程、角色、成果、经历、活动、目标、岗位、分析、时间轴、简历、导入等页面 |
| `src/client/widgets/` | 侧边栏、AI 助手、简历预览、信息卡片等复用组件 |
| `src/client/dialogs/` | 课程、目标、经历、成果、活动、岗位、个人信息等编辑弹窗 |
| `src/client/core/` | 页面壳层、数据刷新、后端运行状态、AI 上下文等核心协调逻辑 |

### 4.3 项目协调与质量控制

组长负责任务拆分、阶段推进、分阶段提交规划和最终材料检查。在开发后期，组长负责统一界面风格、整理运行方式、检查报告内容是否与当前 C++/Qt 项目一致，并修正旧文档中与 Vue/Flask 主架构相关的不一致表述。

## 5. 组员 A 分工说明

### 5.1 API 层与路由开发

组员 A 负责本地 HTTP API 的接口设计与实现，使系统数据可以通过统一的 RESTful 风格接口访问。虽然最终提交形态以 Qt 原生桌面应用为主，但 API 层仍用于模块调试、接口验证和后续扩展。

主要工作包括：

| 工作内容 | 对应模块 |
|---|---|
| 课程、角色、成果、经历、活动、目标、岗位等 API 封装 | `src/api/` |
| HTTP 服务注册与路由组织 | `src/server/` |
| JSON 请求解析与响应生成 | `src/util/JsonUtils*` |
| 接口异常处理与统一响应格式 | API 辅助函数和路由处理逻辑 |

### 5.2 联调贡献

组员 A 参与 API 与 Service 层的联调，确保接口返回的数据结构可以被 Qt 客户端和测试工具稳定使用，并协助验证新增、编辑、删除、查询等基本流程。

## 6. 组员 B 分工说明

### 6.1 数据模型与数据库设计

组员 B 负责核心数据结构和数据库表设计，包括课程、目标、角色、成果、经历、活动、岗位、简历配置、同学对比数据等。该部分主要对应 `src/model/` 和 `resources/schema.sql`。

### 6.2 DAO 与 Service 层实现

组员 B 负责数据访问层和业务逻辑层，使上层 UI 与 API 不直接操作数据库。主要工作包括：

| 层次 | 主要职责 |
|---|---|
| Model 层 | 定义实体字段、序列化和反序列化逻辑 |
| DAO 层 | 封装 SQLite 增删改查操作 |
| Service 层 | 实现 GPA 计算、目标进度、简历素材整合、分析报告、数据导入等业务规则 |

### 6.3 业务功能支撑

组员 B 支撑课程管理、目标追踪、经历归档、成果记录、简历生成、数据分析、时间轴展示和数据导入等功能，为 Qt 页面和 API 层提供统一的数据能力。

## 7. 组员 C 分工说明

### 7.1 测试与运行验证

组员 C 负责测试用例设计、运行验证和问题记录。测试范围包括基础 CRUD 流程、核心接口调用、数据库初始化、应用启动、打包运行、简历导出和 AI 助手可用性检查。

### 7.2 文档与提交材料整理

组员 C 负责整理 README、技术解析文档、期中报告、期末设计报告、运行说明和提交包结构。文档工作重点包括：

| 文档类型 | 说明 |
|---|---|
| 技术解析文档 | 说明各成员负责模块的技术细节 |
| 期中报告 | 说明需求分析、架构设计、分工与进度 |
| 期末设计报告 | 说明最终实现、核心功能、架构模式、测试与总结 |
| 运行说明 | 说明可执行文件和源码运行方式 |

### 7.3 质量检查

组员 C 协助检查文档中的旧表述、路径错误、仓库地址占位符、报告与代码不一致等问题，保证最终材料能够对应当前项目实际。

## 8. 协作流程

团队采用阶段式推进方式完成项目：

| 阶段 | 主要内容 | 主要负责人 |
|---|---|---|
| Stage 1 | 项目初始化、数据库结构、核心模型 | 组长、组员 B |
| Stage 2 | Qt 桌面应用框架、页面壳层、导航结构 | 组长 |
| Stage 3 | 课程、目标、成果、经历等 CRUD 功能 | 组员 B、组员 A |
| Stage 4 | 简历编辑、分析报告、AI 助手、数据导入 | 组长、组员 B |
| Stage 5 | 测试验证、文档完善、提交包整理 | 组员 C、组长 |

## 9. 分工合理性说明

本项目的分工与代码结构基本对应：Qt UI 和核心协调器集中在 `src/client/`，适合由组长统一设计和把控；API 层相对独立，适合组员 A 负责；Model、DAO、Service 是业务功能的基础，适合组员 B 持续维护；测试与文档贯穿项目后期，适合组员 C 进行整理和验证。该分工既能体现团队协作，也能保证项目架构、实现和文档之间保持一致。

## 10. 最终贡献总结

| 成员 | 最终贡献 |
|---|---|
| 组长 | 完成总体架构、Qt 原生 UI、核心协调器、AI 面板、简历编辑器、项目进度和最终一致性检查 |
| 组员 A | 完成 API 层、HTTP 路由、JSON 工具和接口联调 |
| 组员 B | 完成数据模型、DAO、Service、数据库结构和核心业务功能 |
| 组员 C | 完成测试验证、日志检查、技术文档、报告整理和提交材料归档 |
